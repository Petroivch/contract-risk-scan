# Local-first архитектура frontend

## Цель
Сохранить полноценный мобильный UX даже без сети и без обращения пользователя к компьютеру.

## Слой хранения
1. SQLite
   - кэш статусов анализа
   - кэш отчетов
   - кэш истории
   - очередь загрузок
   - таблица schema migrations
2. File cache
   - локальная копия выбранного договора внутри sandbox

## Базовый поток данных
1. Экран вызывает `useApiClient()`.
2. `ApiClientProvider` собирает:
   - HTTP client
   - local stub client
   - SQLiteLocalCache
3. `LocalFirstAdapter` сначала вызывает HTTP client.
4. Если backend отвечает:
   - данные кладутся в SQLite;
   - UI получает реальный backend payload.
5. Если backend недоступен:
   - frontend использует локальный stub client;
   - flow продолжается офлайн;
   - статус, отчет и история все равно попадают в SQLite.

## Offline user flow
1. Пользователь выбирает роль.
2. Пользователь выбирает файл через системный picker.
3. Файл копируется в локальный cache directory.
4. Если upload/analyze через backend невозможен:
   - создаются локальные `contractId` и `analysisId`;
   - анализ продолжается через локальный stub engine;
   - пользователь видит статус, затем отчет;
   - запись попадает в историю.

## Что это дает
- приложение не зависит от компьютера в пользовательском сценарии;
- история и ранее открытые отчеты доступны офлайн;
- новый договор можно проанализировать локально даже без backend.

## Ограничения MVP
- offline fallback сейчас логически эквивалентен локальному demo-analysis, а не полному on-device LLM pipeline;
- автоматическая синхронизация локальных анализов с backend после возвращения сети пока не реализована;
- encryption-at-rest для чувствительных данных остается следующим этапом.
