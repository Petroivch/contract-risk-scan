# Бюджет релиза 228 МБ и вклад DB

## 1. Объект контроля
- Лимит `228 МБ` относится к финальному релизному комплекту.
- Порог задается ключом `build.final_release_size_limit_mb` в таблице `app_config`.

## 2. Вклад DB-части в общий размер
DB-часть включает:
1. Библиотеки хранения, ORM и шифрования.
2. SQL-схему и миграции.
3. Seed/reference-данные.
4. Локальные форматы кэша, если они входят в релизный пакет.

Контрольные ключи:
- `build.db_contribution_target_mb`
- `build.db_contribution_hard_cap_mb`

## 3. Метрики в CI
Обязательные показатели:
1. `total_release_size_mb`
2. `db_contribution_mb`
3. `db_contribution_percent`

Релиз блокируется при превышении `build.final_release_size_limit_mb`.

## 4. Если размер сборки больше 228 МБ
1. Приоритетные меры без функциональной потери:
- убрать dev/test DB-артефакты;
- вычистить неиспользуемые SQL и seed-файлы;
- сделать squash исторических миграций для fresh install.

2. Меры с умеренным компромиссом:
- уменьшить объем prebuilt справочников;
- сократить объем локально хранимой истории;
- ограничить локальный кэш по `mobile.cache.max_contracts`.

3. Архитектурные компромиссы:
- упростить DB SDK или ORM стек;
- сместить часть тяжелого анализа в опциональный серверный путь.

## 5. Меры снижения именно DB-вклада
1. Исключить дублированные enum и константы из кода, хранить их в `app_config` и `language_catalog`.
2. Не включать в mobile package лишние миграции и тестовые фикстуры.
3. Минимизировать seed-данные и держать только необходимое для cold start.
4. Свести локальные индексы к минимуму, необходимому для offline UI.
