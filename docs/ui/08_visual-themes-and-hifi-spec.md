# 08. Hi-fi темы и экранная спецификация

## 1) Общая визуальная система

### Основа
Приложение использует один общий визуальный язык с двумя допустимыми theme-вариантами внутри одной семьи:
- `Editorial Light` — основная тема для MVP;
- `Civic Contrast` — резервный вариант для тестов контраста и platform-tuning.

Обе темы обязаны:
- использовать один и тот же token contract;
- сохранять одинаковую иерархию;
- не менять расположение ключевых действий;
- одинаково работать на `Android` и `iPhone`.

### Структурные принципы
- верх экрана всегда отвечает за контекст;
- центр — за основное действие или чтение;
- низ — за явное действие или навигацию;
- каждая screen composition должна иметь один доминирующий фокус.

## 2) Экранные hi-fi правила

### 2.1 Auth

#### Визуальная задача
Сформировать доверие и ощущение серьезного продукта уже на входе, без маркетингового шума.

#### Композиция
1. верхний editorial block с заголовком, коротким value text и спокойным графическим слоем;
2. плотный form module;
3. один primary CTA.

#### Важные правила
- form module не должен выглядеть как floating promo card;
- ошибка OTP показывается локально, без перерисовки всей сцены;
- returning user path визуально короче и суше, чем first-time path.

### 2.2 Role Selection

#### Визуальная задача
Показать, что роль — это смысловой параметр анализа, а не вспомогательная настройка.

#### Композиция
1. header с кратким объяснением влияния роли;
2. крупный role input block;
3. блок быстрых preset-вариантов;
4. primary CTA.

#### Важные правила
- custom role визуально отличается, но не выглядит как ошибка;
- подсказка должна объяснять, что summary и risk focus зависят от роли;
- primary CTA активен только при валидной роли.

### 2.3 Upload

#### Визуальная задача
Сделать экран похожим на точку юридической загрузки документа, а не на generic file uploader.

#### Композиция
1. header cluster: title + role badge + connection chip;
2. крупный upload panel;
3. metadata area выбранного файла;
4. helper/disclaimer strip;
5. primary CTA row.

#### Важные правила
- upload panel — главный визуальный объект экрана;
- при выбранном файле metadata не прыгает по layout;
- offline queue меняет смысл CTA, но не ломает композицию;
- на small screens header может переноситься в две строки, но не в хаотичный столбец.

#### Кнопки и действия
- tap по upload panel или primary CTA вызывает picker;
- после выбора файла primary CTA становится `start analysis` или queued-action;
- secondary action допускается только если не конкурирует с основным действием.

### 2.4 Analysis Status

#### Визуальная задача
Показать анализ как понятный процесс с этапами, а не как бесконечную крутилку.

#### Композиция
1. header с названием процесса и `RoleBadge`;
2. вертикальный timeline panel;
3. confidence / ETA zone;
4. action row.

#### Важные правила
- активен только один этап одновременно;
- завершенные этапы стабилизируются визуально;
- failed block показывает привязанный recovery action;
- queued и waiting_for_network используют `status.info`, а не error palette.

#### Кнопки и действия
- `Retry` появляется только когда он действительно применим;
- `Resume` должен возвращать queued-flow в processing;
- `Cancel` не должен визуально спорить с primary progress.

### 2.5 Report

#### Визуальная задача
Сделать отчет быстрым для сканирования, но достаточно глубоким для чтения деталей.

#### Композиция
1. summary strip: document title + role badge + confidence chip;
2. tab bar;
3. content zone;
4. action cluster.

#### Вкладка Risks
- `RiskCard` строится вокруг severity rail;
- самые опасные элементы сверху;
- внутри карточки четко отделены: причина риска, влияние, рекомендация;
- `critical` и `high` имеют наиболее сильный контраст и композиционный вес.

#### Вкладка Disputed Clauses
- карточки выглядят аналитично, а не тревожно;
- спорность считывается через warning/neutral логику;
- если disputed clause реально high severity, это отражается только через подтвержденный severity token.

#### Вкладка Summary
- контент разделяется на блоки:
  - обязательства выбранной роли;
  - что эта роль должна другой стороне;
  - что другая сторона должна этой роли;
  - watch-outs и условия, требующие внимания.
- summary должен быть максимально прикладным и читаемым.

#### Кнопки и действия
- tabs реально переключаются;
- change-role action запускает confirm-flow;
- save/share/export деградируют по feature flags без поломки layout;
- section-level retry не сбрасывает весь экран.

### 2.6 History

#### Визуальная задача
Создать ощущение архива проверенных документов, а не бесконечной ленты контента.

#### Композиция
1. header;
2. filter row;
3. ledger-like list;
4. empty/offline states.

#### Важные правила
- каждая карточка должна быстро показывать: документ, дату, severity, роль, cached/queued status;
- cached marker всегда видим;
- queued state визуально отличается от error state.

#### Кнопки и действия
- tap по карточке открывает детали отчета;
- filter chips горизонтально скроллятся;
- offline-доступные карточки не должны выглядеть disabled.

### 2.7 Settings

#### Визуальная задача
Сделать utility-screen таким же аккуратным и взрослым, как core screens.

#### Композиция
1. profile block;
2. language block;
3. local-first controls;
4. lite mode details;
5. legal / sign-out block.

#### Важные правила
- language rows — full-row tap targets;
- длинные локализованные подписи не должны клиповаться;
- destructive actions визуально локализованы внутри своей строки, а не окрашивают весь раздел.

## 3) RoleBadge — hi-fi варианты
- `RoleBadge.compact` — Upload / Status header;
- `RoleBadge.medium` — Report summary strip;
- `RoleBadge.inline` — History row.

Общие правила:
- label берется из данных, не хардкодится;
- custom role получает tokenized variant;
- editable варианты имеют явный affordance;
- history-вариант только информирует.

## 4) Правила для frontend-имплементации
- любые слова вроде `chip`, `rail`, `panel`, `strip` — это структурная роль, а не разрешение invent-ить новый visual family;
- если нужен новый primitive, он сначала добавляется в `03_component-spec.md`;
- frontend должен реализовать плотную вертикальную ритмику, а не увеличивать пустые зоны ради «воздуха»;
- все кнопки должны иметь ясное состояние: default, pressed, disabled, loading.

## 5) Почему этот UI должен выглядеть сильно
- роль видна там, где она реально влияет на смысл;
- анализ показывается как процесс, а не как черный ящик;
- report визуально ранжирует опасность, а не просто перечисляет данные;
- history выглядит как реестр, что усиливает доверие;
- даже после упрощения ради size budget структура остается узнаваемой.

## 6) Что frontend не должен упростить
- нельзя превращать status screen в один loader;
- нельзя прятать роль глубоко в metadata;
- нельзя делать одинаковый visual weight для `critical` и `low`;
- нельзя строить report как бесконечную плоскую колонку без смысловых слоев;
- нельзя жертвовать читабельностью ради декоративных градиентов.
